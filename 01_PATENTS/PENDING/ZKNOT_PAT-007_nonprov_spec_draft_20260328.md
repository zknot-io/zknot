## BACKGROUND OF THE INVENTION

### Field of the Invention

The present invention relates to evidence attestation systems, and more particularly to systems and methods for producing a single cryptographic record that simultaneously proves both (1) that a specific device was powered without any data transfer and (2) that a specific human approved a specific evidence hash using a user-actuated hardware signing device, the two proofs being cryptographically bound by a shared session identifier within a single append-only hash-chained ledger entry.

### Description of Related Art

Evidence custody in forensic, election auditing, pharmaceutical, and regulatory contexts requires proof of multiple concurrent facts: that a device was powered, that data was or was not transferred, that a specific person authorized a specific action, and that these events occurred in a defined sequence. Current approaches treat these as separate, independently documented events:

- Power events are documented by procedural logs, timestamps in device firmware, or system event records
- Chain-of-custody custody transfers are documented by witness signatures, numbered seals, and paper forms
- Digital signing events are recorded by PKI certificates, code-signing logs, or hardware token audit logs

None of these approaches produces a single cryptographic artifact that proves all three facts simultaneously:
1. A device was powered without data exchange (physical fact)
2. A human reviewed and specifically approved the evidence hash (behavioral fact)
3. Both facts occurred within the same defined custody session (temporal binding)

The absence of such a unified proof creates gaps exploitable by opposing counsel: even if power and signing events are separately documented, the link between them rests on procedural assertion rather than cryptographic proof. An attorney can challenge whether the power event and the signing event refer to the same physical device, the same evidence, and the same time.

## SUMMARY OF THE INVENTION

The present invention provides methods and systems for producing a CombinedSession attestation record that cryptographically binds a power attestation record (from a physical data-path elimination device) with a signing attestation record (from a user-actuated hardware signing device) using a shared session identifier, and records both as a single entry in an append-only hash-chained ledger.

In one aspect, the invention provides a method comprising: generating a power session attestation record by a power verification device wherein data conductors are physically absent; generating a signing attestation record by a user-actuated signing device; binding the two records by a shared session identifier; and recording a combined entry in an append-only hash-chained ledger, the combined entry comprising both records and proving both physical data-transfer impossibility and human-gated evidence authorization within a single custody event.

## DETAILED DESCRIPTION OF THE INVENTION

### 1. The CombinedSession Concept

A CombinedSession attestation record is a single chain entry in the ZK-LocalChain ledger that binds two hardware-attested sub-records:

**Sub-record A — PowerSession:** Generated by a PowerVerify device (PAT-002) when a device under attestation is connected to power. Proves that the device received electrical power via a USB path in which D+ and D- data conductors are physically absent, making data transfer impossible by physical construction.

**Sub-record B — ZKeySign:** Generated by a ZKKey device (PAT-001) when a user views the SHA-256 hash of the evidence and physically actuates the hardware button, causing the ATECC608B hardware secure element to generate an ECDSA signature. Proves that a specific human approved a specific evidence hash using a specific hardware device.

The two sub-records are bound by a shared `session_id`: a UUID generated at the start of the custody session and included in both attestation artifacts. The CombinedSession chain entry in the ledger records both sub-records under the shared session_id, proving both events are part of the same custody session.

### 2. Session Identifier Distribution

The session_id is generated by the evidence management system at the start of each custody session and communicated to both the PowerVerify device and the ZKKey device by the session controller. This distribution may occur via:

- **USB configuration packet:** The session controller transmits the session_id to each device via the USB configuration interface before the custody event begins
- **QR code display:** The session controller displays a QR code encoding the session_id, which both devices scan optically (applicable to the ZKKey Air variant)
- **Pre-provisioned:** The session_id is pre-generated and stored in both devices before field deployment for a specific custody event

### 3. Combined Entry Record Format

The CombinedSession chain entry in the ledger has `artifact_type = COMBINED_SESSION` and comprises:

```json
{
  "artifact_type": "COMBINED_SESSION",
  "session_id": "<shared UUID>",
  "power_record": {
    "artifact_id": "<UUID>",
    "artifact_type": "POWER_SESSION",
    "device_id": "<PowerVerify serial number>",
    "signature": "<ECDSA signature from PowerVerify HSE>",
    "public_key": "<PowerVerify public key>",
    "signed_at": "<ISO 8601 timestamp>"
  },
  "signing_record": {
    "artifact_id": "<UUID>",
    "artifact_type": "ZKEY_SIGN",
    "device_id": "<ZKKey serial number>",
    "challenge_hash": "<SHA-256 of evidence>",
    "signature": "<ECDSA signature from ZKKey HSE>",
    "public_key": "<ZKKey public key>",
    "signed_at": "<ISO 8601 timestamp>"
  },
  "prev_hash": "<hash of preceding ledger entry>",
  "entry_hash": "<hash of this combined entry>",
  "chain_position": "<integer>",
  "short_code": "<human-readable code>"
}
```

### 4. What the Combined Record Proves

A CombinedSession chain entry provides independently verifiable proof of all of the following facts:

1. **Physical data impossibility:** A specific PowerVerify device (identified by `power_record.device_id`) was used during this session. Any party may independently verify by continuity-testing the identified PowerVerify device to confirm D+ and D- conductors are absent.

2. **Human-gated authorization:** A specific ZKKey device (identified by `signing_record.device_id`) generated a signature, which required a human to view the challenge hash on the OLED display and press the physical button.

3. **Evidence binding:** The `signing_record.challenge_hash` field contains the SHA-256 hash of the specific evidence data. Any party can independently verify that the signed hash matches the hash of the evidence in question.

4. **Session binding:** Both sub-records carry the same `session_id`, proving they are part of the same defined custody session rather than two unrelated events assembled retroactively.

5. **Chain integrity:** The `prev_hash` and `entry_hash` fields link the combined record into the append-only hash chain, proving it was not inserted retroactively.

6. **Hardware authentication:** Both ECDSA signatures are verifiable against the respective device public keys, which are bound to the device serial numbers registered at manufacture. Any party can verify the signatures using standard ECDSA verification without vendor involvement.

### 5. Verification

A CombinedSession chain entry is publicly verifiable via the short code at verifyknot.io/{short_code} without account creation, login, or vendor contact. The verification display shows all six categories of proof listed above in a human-readable format accessible to non-technical fact-finders including attorneys, judges, auditors, and regulators.

### 6. Temporal Proof

The `signed_at` fields of both sub-records are generated by the respective hardware devices at the time of the custody event. The timestamps provide temporal evidence. In embodiments where the devices are synchronized to an authoritative time source (NTP, GPS, or hardware RTC with documented calibration), the timestamps constitute independently verifiable temporal proof.

### 7. Application to the EvidenceProtocol Workflow

The CombinedSession is the culminating record of the ZKNOT EvidenceProtocol workflow:

1. PowerVerify connected → `power_record` generated (device powered without data exchange)
2. Evidence collected or sealed → hash computed
3. ZKKey receives evidence hash as challenge → displays hash → user actuates → `signing_record` generated
4. CombinedSession entry recorded in ZK-LocalChain → `COMBINED_SESSION` chain entry with both sub-records and shared `session_id`
5. Short code printed or transmitted → publicly verifiable at verifyknot.io

This workflow produces a single entry that closes the evidentiary loop: a human approved a specific evidence hash, a specific device was powered without data exchange, and both events are cryptographically proven to be part of the same custody session.

## CLAIMS

**Claim 1.** A method for producing a combined hardware-attested custody record comprising:
generating a power attestation record by a first hardware device, the first hardware device comprising VBUS and GND conductors and physically absent D+ and D- data conductors, the power attestation record comprising a first session identifier and a first digital signature generated by a first hardware secure element of the first hardware device;
generating a signing attestation record by a second hardware device, the second hardware device comprising a hardware secure element, an integrated display, and a physical input element, the signing attestation record comprising the first session identifier and a second digital signature generated by a second hardware secure element of the second hardware device after a user has viewed a challenge hash on the integrated display and physically actuated the physical input element;
appending a combined chain entry to an append-only hash-chained ledger, the combined chain entry comprising both the power attestation record and the signing attestation record bound by the shared first session identifier; and
wherein the combined chain entry provides cryptographic proof that a specific device was powered without data transfer and a specific user approved a specific challenge hash within the same custody session.

**Claim 2.** The method of claim 1, wherein the combined chain entry further comprises a preceding entry hash field containing the hash of the immediately preceding entry in the append-only hash-chained ledger, such that any modification to the combined chain entry is detectable by hash recomputation.

**Claim 3.** The method of claim 1, wherein appending the combined chain entry does not require network connectivity.

**Claim 4.** The method of claim 1, further comprising generating a human-readable short code derived deterministically from at least one of the first digital signature and the second digital signature, wherein the short code is resolvable at a public verification service to display the combined chain entry without account credentials.

**Claim 5.** The method of claim 1, wherein the first session identifier is generated by a session controller and transmitted to both the first hardware device and the second hardware device before the custody session begins.

**Claim 6.** The method of claim 1, wherein the combined chain entry is independently verifiable by any party with access to an exported version of the append-only hash-chained ledger by: verifying the first digital signature against the public key of the first hardware device; verifying the second digital signature against the public key of the second hardware device; and recomputing the chain entry hash to confirm ledger integrity.

**Claim 7.** The method of claim 1, wherein the absence of D+ and D- conductors in the first hardware device is independently verifiable by electrical continuity testing of the first hardware device.

**Claim 8.** A system for combined hardware-attested evidence custody comprising:
a power verification device comprising VBUS and GND conductors and physically absent D+ and D- conductors, the power verification device configured to generate a power attestation record comprising a session identifier and a digital signature from a first hardware secure element;
a user-actuated signing device comprising a hardware secure element, an integrated display, and a physical input element, the user-actuated signing device configured to generate a signing attestation record comprising the session identifier and a digital signature from a second hardware secure element only upon physical actuation of the physical input element after the integrated display has shown a challenge hash; and
an append-only hash-chained ledger configured to record a combined entry comprising both the power attestation record and the signing attestation record bound by the shared session identifier.

**Claim 9.** The system of claim 8, further comprising a public verification service configured to resolve a human-readable code to a display of the combined entry without requiring account credentials, vendor access, or network access to the power verification device or the user-actuated signing device.

**Claim 10.** The system of claim 8, wherein the append-only hash-chained ledger operates without network connectivity.

**Claim 11.** The system of claim 8, wherein the session identifier is a UUID generated by a session controller and communicated to both the power verification device and the user-actuated signing device before commencement of the custody session.

**Claim 12.** A combined hardware-attested custody record comprising:
a power attestation sub-record comprising a session identifier, a power device identifier, a timestamp, and a first digital signature generated by a first hardware secure element in a device having physically absent data conductors;
a signing attestation sub-record comprising the session identifier, a signing device identifier, a challenge hash, a timestamp, and a second digital signature generated by a second hardware secure element in a user-actuated signing device after display of the challenge hash to a user and physical actuation of a dedicated input element by the user;
wherein the power attestation sub-record and the signing attestation sub-record are bound by the shared session identifier in a single chain entry of an append-only hash-chained ledger; and
wherein the single chain entry is independently verifiable to prove simultaneously that a device was powered without data transfer and a user specifically authorized a specific evidence hash within the same custody session.

**Claim 13.** The combined hardware-attested custody record of claim 12, further comprising a chain entry hash computed over the fields of the combined record including a preceding entry hash, such that the record is linked to an append-only hash chain.

**Claim 14.** The combined hardware-attested custody record of claim 12, further comprising a human-readable short code derived deterministically from at least one of the first or second digital signature, the short code being resolvable to a public display of the combined record without account credentials.

**Claim 15.** A method for independent verification of a combined hardware-attested custody record comprising:
receiving an exported append-only hash-chained ledger comprising at least one combined chain entry, the combined chain entry comprising a power attestation sub-record and a signing attestation sub-record sharing a session identifier;
verifying a first digital signature in the power attestation sub-record against a first device public key using elliptic curve digital signature algorithm verification;
verifying a second digital signature in the signing attestation sub-record against a second device public key using elliptic curve digital signature algorithm verification;
recomputing the chain entry hash of the combined chain entry and confirming it matches the preceding hash field of the subsequent chain entry; and
determining that the combined chain entry proves both physical data-transfer impossibility and human-gated authorization within the same custody session;
wherein all verification steps are performed without network access, vendor involvement, or access to the original signing hardware.

## ABSTRACT

A combined hardware-attested custody record binds two independent hardware attestation events — a power attestation from a device with physically absent data conductors and a signing attestation from a user-actuated hardware signing device — into a single chain entry in an append-only hash-chained ledger. The two attestation records share a session identifier that cryptographically proves they are part of the same custody session. The combined entry provides independent verifiable proof that a specific device was powered without data exchange, a specific human viewed and approved a specific evidence hash, and both events occurred within the same defined custody session. The combined entry is recorded in an offline-capable ledger, includes a preceding entry hash linking it to the chain, and is resolvable via a human-readable short code at a public verification service without account credentials or vendor involvement.

What is ZKNOT™
6 questions
What does ZKNOT™ actually do?

ZKNOT™ builds hardware and software that makes evidence self-proving. Every custody event — connecting a device, signing an artifact, applying a seal, transferring custody — is recorded as a cryptographically unforgeable chain entry that any party can independently verify without contacting ZKNOT™.

The result: a chain of custody where the math and physics do the testifying, not the operator. Any party with a short code can verify at verifyknot.io in seconds.

Is ZKNOT™ a security company?

No. ZKNOT™ is an evidence company. The distinction matters.

Security companies prevent bad things from happening. Evidence companies prove what happened — and when, by whom, and in what state. ZKNOT™ products are designed so that the chain of custody is self-proving regardless of whether any security control worked correctly. Even if a device was compromised, the ZKNOT™ chain record shows exactly what was attested and when.

What problem does ZKNOT™ solve that existing tools don't?

Existing chain-of-custody tools enforce constraints through software policy — write-blockers that run firmware, databases that restrict edits, access controls that prevent deletion. The problem: any software control can be misconfigured, exploited, or overridden by someone with sufficient access. In court, this means the examiner testifies to a software state. Defense challenges it.

ZKNOT™ enforces constraints at two levels that cannot be overridden:

Physics — PowerVerify™ severs the write data path at the PCB level. No firmware, no configuration, no administrator can restore it. Testable with a multimeter.
Mathematics — ZK-LocalChain™ hash-chains every entry. Retroactive modification is computationally impossible — detectable by any party with two consecutive entries.

The verification is also structural: anyone verifies at verifyknot.io without vendor involvement, eliminating the conflict-of-interest argument.

Does ZKNOT™ use blockchain?

No. ZK-LocalChain™ is a hash-chained ledger — it uses the same underlying cryptographic principle (SHA-256 chain linkage) without any of the blockchain infrastructure: no distributed network, no consensus mechanism, no tokens, no miners.

The result is an append-only ledger that is faster, simpler, fully offline-capable, and independently auditable. A single exported JSON file contains everything needed to verify the entire chain — no node required, no network call, no third party.

The chain is independently verifiable from any exported entry. You do not need ZKNOT™ infrastructure, a full node, or network access to verify a record.
What does "independently verifiable" actually mean?

It means any party — opposing counsel, a judge, a regulatory auditor, a journalist — can verify a ZKNOT™ chain record without:

How It Works
5 questions
What is EvidenceProtocol™ and how does it work end to end?

EvidenceProtocol™ is the end-to-end ZKNOT™ workflow. Six steps, each enforced by a different mechanism:

01 PowerVerify™ connect — write data path severed at hardware level before any software runs
02 Forensic capture — image using any standard tool; hardware enforces write protection below the tool layer
03 TrustSeal™ application — physical seal applied and simultaneously bound to a chain entry; atomic event
04 ZKKey™ sign — operator presses physical button; ECDSA P-256 signature produced by secure element
05 Chain record written — entry written to ZK-LocalChain™, hash-linked to all prior entries
06 PDF report exported — signed chain-of-custody report with QR codes linking to verifyknot.io for each entry

Full details at zknot.io/protocol.

What is a short code and where does it appear?

A short code is a human-readable identifier assigned to every chain entry at write time. Format: ZK-XXXX-XXX. It appears:

Printed on every TrustSeal™ label
Displayed on the ZKKey™ OLED screen after signing
Included in every PDF chain-of-custody report as text and QR code
Returned by the API on every attestation event

Anyone with the short code can resolve the full chain entry at verifyknot.io — no account, no login.

How does the hash chain work? What does it actually prove?

Each chain entry contains a prev_hash field — the SHA-256 hash of the immediately preceding entry. This means every entry is mathematically bound to every entry before it.

What this proves: if anyone modifies any prior entry, the prev_hash in the next entry no longer matches. The modification is detectable by any party with two consecutive entries — no ZKNOT™ infrastructure required. The detection is mathematical, not institutional.

To verify chain integrity: compute SHA-256 of entry N (excluding its own entry_hash field) and check it matches prev_hash in entry N+1. Two entries, a SHA-256 library, and the math proves the chain is intact.
What happens to the chain if ZKNOT™ goes offline or ceases to exist?

Nothing happens to records that have already been attested. Every chain entry is self-contained in the exported JSON artifact — it includes the full attestation data, the device public key, and the ECDSA signature. Any party can verify any record independently using a SHA-256 library and an ECDSA verification function.

ZKNOT™ infrastructure (api.zknot.io, verifyknot.io) provides convenience — fast lookup, public resolution, PDF export. It does not provide the proof. The proof is in the mathematics of the chain entry itself.

Design principle: the chain is verifiable by anyone, from any exported entry, without ZKNOT™. This is not a feature — it is a structural guarantee.
Can a chain entry be deleted or modified after the fact?

No delete operation exists at any layer of the stack. ZK-LocalChain™ is append-only — the ledger format has no delete command. The database schema has no delete operation. The API has no delete endpoint.

Modification: any change to a chain entry invalidates its entry_hash, which breaks the prev_hash of every subsequent entry. Silent modification is mathematically impossible — it is detectable from any two consecutive entries.

This is not a policy restriction. It is a structural property of the system.

Products
6 questions
What is ZKKey™ and what makes it different from a software signing tool?

ZKKey™ is a hardware signing device built on an STM32F411 microcontroller and a Microchip ATECC608B secure element. The private ECDSA P-256 key is generated inside the secure element and cannot be exported — by anyone, under any condition, including ZKNOT™.

What makes it structurally different from software signing:

Physical presence required — only a button press on the physical device produces a signature. No remote signing, no scripted automation, no administrator override.
Post-nonce enforcement — after the session nonce expires, the signing circuit is physically open. No software command can re-enable it for that session.
Hardware key isolation — the ATECC608B zone-locks the key at provisioning. Export is prohibited at the silicon level.
How does PowerVerify™ differ from a software write-blocker?

Software write-blockers intercept write commands in the firmware or driver layer and deny them. PowerVerify™ does not intercept anything — it physically severs the write data lines at the PCB trace level. There is no firmware path, no driver to exploit, no configuration to misconfigure.

The critical difference in court: a software write-blocker requires the examiner to testify that it was correctly configured and free from vulnerabilities. PowerVerify™ requires only a continuity test — a multimeter measurement that any party can perform, in any courtroom, with no technical expertise required.

Defense argument against software write-blockers: "How do we know the firmware had no vulnerability that permitted writes?" — This argument does not apply to a severed PCB trace.
What artifact types does ZK-LocalChain™ support?

Five artifact types are defined in schema v1:

COMBINED_SESSION — full end-to-end session attestation combining PowerVerify and ZKKey events
POWER_SESSION — PowerVerify™ device connection and write-path attestation event
ZKEY_SIGN — individual ZKKey™ hardware signing event
TRUST_SEAL — TrustSeal™ physical seal application event, bound to artifact hash
CHAIN_EXPORT — signed PDF report export event

All types follow the same attestation artifact JSON schema v1. See docs for the full field reference.

How does TrustSeal™ bind a physical item to the digital chain?

The binding event is atomic — the seal scan and the chain write happen simultaneously. When the barcode on the TrustSeal™ is scanned, the ZKKey™ immediately writes a TRUST_SEAL chain entry containing the seal's unique ID, the artifact hash, the operator device ID, and the timestamp. There is no window between physical application and digital record.

The seal itself carries the short code, printed and encoded as a QR code. If the seal is removed, a permanent VOID pattern appears on both the item surface and the seal backing. The chain entry persists regardless — proving the seal existed, when it was applied, and who applied it.

Can ZKNOT™ products work together, or do I need to use all of them?

Products are modular. Each enforces a specific constraint and can be used independently or in combination. Common deployments:

ZKKey™ + ZK-LocalChain™ only — document attestation and chain record for journalism or legal use cases
PowerVerify™ + ZKKey™ + ZK-LocalChain™ — full digital forensics acquisition workflow
ZKKey™ + TrustSeal™ + ZK-LocalChain™ — physical item custody with cryptographic binding
Full EvidenceProtocol™ — all four products in the complete six-step workflow

Contact ops@zknot.io to scope the right configuration for your use case.

What happens if a ZKKey™ device is lost or stolen?

Each ZKKey™ has a unique device ID and corresponding ECDSA public key registered in the chain. If a device is lost or stolen:

The device can be revoked in the registry — all future signatures from that device ID are flagged
Prior chain entries signed by the device remain valid — the signatures are mathematically sound
The attacker cannot extract the private key from the ATECC608B — the key is hardware-locked and non-exportable
Any signature produced by the stolen device still requires a physical button press — remote or automated signing is structurally impossible

Can I verify a record without using verifyknot.io?

Yes. The API is public and unauthenticated for verification:

GET https://api.zknot.io/v1/verify/{short_code}

Returns the full attestation artifact JSON including device public key and ECDSA signature. No API key required.

From there, independent cryptographic verification requires only a SHA-256 library and an ECDSA P-256 verification function — both available in every major language. See docs for the verification algorithm.

What does "VERIFIED — CHAIN INTACT" mean vs a failure state?

VERIFIED — CHAIN INTACT means three things checked out on lookup:

The entry hash matches — the entry data hasn't been modified since writing
The ECDSA signature verifies — the registered ZKKey™ device produced this signature
The chain linkage is intact — prev_hash matches the prior entry's entry_hash

A failure state (CHAIN_INTEGRITY_FAILURE or SIGNATURE_INVALID) would indicate tampering — something that should never occur in normal operation. If you ever see one, contact ops@zknot.io immediately.

Is verification access logged? Who can see what records I look up?

The verification API is public and unauthenticated — no login, no account. Standard server logs may record the request IP and short code accessed, consistent with normal web server logging. ZKNOT™ does not sell, share, or analyze verification lookup data for commercial purposes.

For sensitive operational use cases requiring private verification (where the short code itself should not be disclosed to ZKNOT™ servers), contact ops@zknot.io to discuss on-premise or air-gapped verification options.

Security & Cryptography
5 questions
What cryptographic algorithms does ZKNOT™ use?
ECDSA P-256 — device signing, per NIST FIPS 186-4. Keys generated and stored in ATECC608B secure element.
SHA-256 — artifact hashing and chain linkage. Per NIST FIPS 180-4.
Hardware TRNG — nonce generation uses the ATECC608B's hardware true random number generator.

All algorithms are NIST-approved and suitable for federal use. No proprietary or unreviewed cryptography is used anywhere in the stack.

Can ZKNOT™ staff access or read my chain records?

Chain records written to the hosted ZK-LocalChain™ are stored on ZKNOT™ infrastructure and are accessible to ZKNOT™ staff with database access. This is consistent with any SaaS provider.

For deployments where chain records must be fully private and ZKNOT™ must have zero access, ZK-LocalChain™ supports on-premise deployment — the ledger runs entirely on your hardware with no data leaving your environment. Contact ops@zknot.io to discuss on-premise licensing.

Key point: even if ZKNOT™ staff accessed your records, they could not alter them without breaking the hash chain — which would be detectable by you from any exported entry.
Has ZKNOT™ been independently audited?

ZKNOT™ is an early-stage company and has not yet undergone a formal third-party security audit. The cryptographic algorithms (ECDSA P-256, SHA-256) are NIST-approved and extensively reviewed. The ATECC608B secure element is a widely-deployed commercial component used in production security systems globally.

We welcome and actively seek independent technical review. Researchers and potential government customers interested in technical evaluation should contact ops@zknot.io.

Is ZKNOT™ FIPS 140-2 certified?

Not currently. The ATECC608B secure element is a FIPS 140-2 candidate-class component — it implements NIST-approved algorithms (ECDSA P-256, SHA-256, HMAC) with hardware key isolation. Full FIPS 140-2 validation of the ZKNOT™ system as a whole is on the product roadmap, prioritized for federal procurement opportunities.

ZKNOT™ products use standard commercial cryptography (ECDSA P-256, SHA-256) implemented in a commercially available secure element (Microchip ATECC608B). What does the ZKNOT™ API look like?

REST API hosted at api.zknot.io. Two core endpoints in production:

POST /v1/attest — ingest a signed artifact and write to chain

GET /v1/verify/{short_code} — resolve any short code to a verified chain entry (public, no auth)

Full API reference including request/response schemas, error codes, and SDK examples at zknot.io/docs.

Can ZKNOT™ integrate with existing forensic tools like FTK or Cellebrite?

Yes. PowerVerify™ operates below the software layer — it works with any imaging tool without modification. FTK Imager, Cellebrite, dd, custom tools — all work identically because PowerVerify™ enforces write protection at the hardware level before any software runs.

ZK-LocalChain™ integrates via API. A thin SDK wrapper can post attestation events from any forensic workflow that supports outbound API calls or scripted post-acquisition steps. Contact ops@zknot.io to discuss specific integration requirements.

Offline & Field Operations
3 questions
Does ZKNOT™ work without an internet connection?

Yes. ZK-LocalChain™ is offline-capable by design. The full attestation workflow — PowerVerify™ connect, ZKKey™ sign, TrustSeal™ apply, chain write — operates entirely on local hardware with no network dependency. Chain entries are stored locally and sync to api.zknot.io when connectivity is available.

This is not a degraded mode. Offline operation is a first-class deployment scenario specifically designed for forward-deployed, field, and secure-facility use.

ZKNOT™ does not use the term "air-gapped." The correct term is offline-capable or intermittently-connected — the system is designed for environments where network connectivity is unreliable or unavailable, not permanently isolated.
How does chain sync work when connectivity is restored?

When the ZKKey™ or local ZK-LocalChain™ instance regains connectivity, it posts pending chain entries to api.zknot.io via POST /v1/attest in sequence order. Chain position and hash linkage are preserved exactly — the sync does not modify any entry. After sync, all entries are publicly resolvable at verifyknot.io.

Entries created offline are fully valid before sync — chain integrity can be verified locally from the device export at any time without network access.

What are the hardware requirements for field deployment?
ZKKey™ — compact handheld device, USB-A powered from any host, no external power required
PowerVerify™ — passive passthrough, bus-powered, no external power
Host device — any laptop or workstation with a USB-A port and Python or the ZKKey™ client software
TrustSeal™ scanner — USB HID barcode scanner (DE2120), plug-and-play, no driver install

The full field kit fits in a standard evidence bag. No rack hardware, no server, no dedicated workstation required.

Legal & Court Admissibility
3 questions
Is ZKNOT™ evidence admissible in court?

ZKNOT™ is not a law firm and cannot provide legal advice on admissibility. Admissibility depends on jurisdiction, court rules, and the specific facts of the case.

What we can say is that ZKNOT™ records are designed to satisfy the technical foundations that courts use to evaluate digital evidence integrity: chain of custody documentation, write-protection verification, hash-based integrity proof, and timestamp attestation. The PowerVerify™ continuity test provides a physically verifiable fact that does not require expert testimony to explain.

Consult qualified legal counsel for guidance on admissibility in your specific jurisdiction and case.
Does using ZKNOT™ require an expert witness to testify?

That is a decision for counsel, but ZKNOT™ is specifically designed to reduce dependence on expert testimony by making the proof physical and mathematical rather than institutional.

PowerVerify™ — write protection is a continuity test. Any witness can perform it. No cryptography expertise required.
Hash verification — SHA-256 is a publicly documented, court-accepted algorithm. The comparison is deterministic: hash matches or it doesn't.
verifyknot.io — opposing counsel can verify the record themselves, in the courtroom, on their phone.

ZKNOT™ staff can provide technical declarations or expert testimony when required. Contact ops@zknot.io.

What happens if ZKNOT™ is subpoenaed for chain records?

ZKNOT™ will comply with valid legal process. For customers with on-premise deployment, ZKNOT™ has no access to chain records and cannot be compelled to produce records it does not hold.

For cloud-hosted chains, ZKNOT™ maintains records in accordance with our data retention policy. Customers requiring zero third-party access to chain data should use on-premise deployment. Contact ops@zknot.io for details.
